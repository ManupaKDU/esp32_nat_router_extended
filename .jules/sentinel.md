## 2025-02-28 - Timing Attack in Password Check
**Vulnerability:** The lock handler used `strcmp` to compare passwords. This allows an attacker to deduce the password character by character by measuring the time it takes for the comparison to fail.
**Learning:** Standard C string comparison functions like `strcmp` short-circuit and return immediately upon finding a difference. This creates a timing side-channel.
**Prevention:** Sensitive data like passwords or tokens should always be compared using a constant-time comparison function, such as `crypto_memcmp`. First check that the lengths are equal (to prevent other types of attacks), and then use `crypto_memcmp` to check the contents.
## 2025-02-28 - DoS via Memory Leak in HTTP Handlers
**Vulnerability:** HTTP GET handlers (`index_get_handler` and `lock_handler`) were requesting string configurations (e.g., `lock_pass`, `sta_identity`) using `get_config_param_str` and `get_config_param_blob`, but never freeing the allocated memory. This resulted in a heap memory leak on every request.
**Learning:** `get_config_param_str` and `get_config_param_blob` allocate memory dynamically on the heap via `malloc`. Because this is an embedded ESP32 context where RAM is heavily constrained, even a small leak per request rapidly exhausts the heap, causing a severe Denial of Service (DoS) vulnerability.
**Prevention:** Always verify the caller's memory ownership responsibility when using configuration retrieval functions. Ensure that any `malloc`ed configuration strings or blobs are appropriately `free()`d within the request handler. When pointer reassignment happens (e.g. `sta_identity` assigned to `""`), safely save the original allocated address to be freed later.
