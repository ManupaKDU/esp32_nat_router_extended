## 2024-05-01 - Prevent Credential Exposure in Logs (CWE-532)
**Vulnerability:** Plaintext passwords (`pass2Param`) and user credentials (`set_sta_ent_arg.user`) were being printed to logs via `ESP_LOGI` formatting.
**Learning:** Even internal logging mechanisms like `ESP_LOGI` can act as sinks for credential exposure, which breaks the required security posture by leaking sensitive configuration values to UART/console.
**Prevention:** Explicitly redact values for keys containing keywords such as `pass`, `user`, `unlock`, or `identity` before they hit the logging functions. Replace the format string variable with static `***REDACTED***` placeholders.
## 2024-05-01 - Avoid Double Free masquerading as Memory Exhaustion (CWE-415)
**Vulnerability:** A double-free vulnerability in `lockhandler.c` (`unlock_handler`) was caused by prematurely freeing a heap-allocated buffer (`unlockParam`) inside an `if` block, and then blindly freeing it again at the end of the function. This heap corruption results in a router panic/reboot, which acts as a Denial of Service.
**Learning:** In C request handlers handling multiple conditional branches, memory cleanup must be strictly centralized at the end of the function's scope to prevent double frees, unless variables are explicitly set to `NULL` after the first `free`. Moreover, misleading error logs (e.g., `ESP_LOGE(TAG, "Memory allocation failed...")` on empty parameter branches) obscured failed authentication attempts, making debugging and auditing harder.
**Prevention:** Avoid inline `free()` calls inside conditional branches if the same pointer is freed at function exit. Use centralized exit labels or structured cleanup. Ensure error logs accurately reflect business logic failures (like empty or mismatched parameters) rather than blindly emitting system-level error messages (like `malloc` failures).
