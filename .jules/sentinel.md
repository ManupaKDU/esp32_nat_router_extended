## 2024-05-01 - Prevent Credential Exposure in Logs (CWE-532)
**Vulnerability:** Plaintext passwords (`pass2Param`) and user credentials (`set_sta_ent_arg.user`) were being printed to logs via `ESP_LOGI` formatting.
**Learning:** Even internal logging mechanisms like `ESP_LOGI` can act as sinks for credential exposure, which breaks the required security posture by leaking sensitive configuration values to UART/console.
**Prevention:** Explicitly redact values for keys containing keywords such as `pass`, `user`, `unlock`, or `identity` before they hit the logging functions. Replace the format string variable with static `***REDACTED***` placeholders.## 2024-05-15 - Prevent Stack Buffer Overflow via VLAs and strcat (CWE-120)
**Vulnerability:** In `portmaphandler.c`, a Variable Length Array (VLA) `char resultIP[strlen(defaultIP) + strlen(param) + 2];` was being initialized using an untrusted user-controlled POST parameter `param` (representing the IP address). The subsequent code used `strcat` to append this untrusted input. If an attacker provided a massive string for `param`, it would cause a stack overflow, leading to memory corruption or a denial-of-service (DoS) crash before the logic could reach IP validation (`ipaddr_addr`).
**Learning:** Avoid using Variable Length Arrays (VLAs), especially when their length is derived from externally controlled inputs, as they reside on the stack which has limited size. Furthermore, `strcat` provides no bounds checking.
**Prevention:** Use fixed-size arrays based on the maximum expected bounds of the data (e.g., maximum length of an IPv4 string plus slack for the input parameter), and use bounds-checked functions like `snprintf` to construct the string safely, allowing it to truncate rather than overflow.
