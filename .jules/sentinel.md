## 2024-05-01 - Prevent Credential Exposure in Logs (CWE-532)
**Vulnerability:** Plaintext passwords (`pass2Param`) and user credentials (`set_sta_ent_arg.user`) were being printed to logs via `ESP_LOGI` formatting.
**Learning:** Even internal logging mechanisms like `ESP_LOGI` can act as sinks for credential exposure, which breaks the required security posture by leaking sensitive configuration values to UART/console.
**Prevention:** Explicitly redact values for keys containing keywords such as `pass`, `user`, `unlock`, or `identity` before they hit the logging functions. Replace the format string variable with static `***REDACTED***` placeholders.## 2024-05-02 - Prevent Buffer Overflow with Variable Length Arrays and strcat
**Vulnerability:** Constructing IP addresses in `src/urihandler/portmaphandler.c` using Variable Length Arrays (`char resultIP[strlen(defaultIP) + strlen(param) + 2]`) and `strcat(resultIP, param)` creates a potential stack buffer overflow. A maliciously crafted or unexpectedly long `param` could overflow the stack, as VLA bounds rely on dynamic lengths that might not correctly account for total memory constraints or subsequent unconstrained `strcat` operations.
**Learning:** Combining VLAs with unsafe string functions like `strcat` is highly risky, especially when parsing network-provided inputs (`param` from `readUrlParameterIntoBuffer`). The buffer size could theoretically exceed available stack space, and `strcat` does not enforce bounds checking.
**Prevention:** Always use fixed-size stack buffers for inherently bounded data (like IP addresses, which are max 15 chars + null terminator = 16 bytes). Replace unconstrained concatenation (`strcat`) with `snprintf(buf, sizeof(buf), ...)` to guarantee bounds checking and prevent memory corruption.
