## 2024-03-20 - Prevent Credential Leakage in Application Logs
**Vulnerability:** Plaintext Wi-Fi passwords, UI lock passwords, and WPA2 enterprise identities/usernames were being exposed in system logs via `ESP_LOGI` statements across multiple files (`esp32_nat_router.c`, `http_server.c`, `applyhandler.c`).
**Learning:** These logging statements were likely added for debugging purposes during development but left in production code. Since this firmware runs on a device that may output logs over serial or to external services, exposing credentials in plaintext represents a critical security risk.
**Prevention:** Never log sensitive data (passwords, tokens, usernames) in plaintext. When logging connection attempts or configuration updates, either omit the sensitive field entirely or redact it (e.g., using `***`). Regularly audit `ESP_LOG` statements for potential credential leakage.
## 2023-10-24 - Unbounded Variable Length Array (VLA) Stack Overflow via Content-Length
**Vulnerability:** Several HTTP request handlers allocated buffers on the stack using Variable Length Arrays (VLAs) with a size strictly derived from the client-supplied `Content-Length` header (e.g., `char buf[req->content_len + 1]`). This poses a critical risk of stack exhaustion/overflow on constrained ESP32 devices when handling malicious requests with extremely large lengths.
**Learning:** In HTTP/TCP handlers on ESP-IDF, `req->content_len` is an untrusted, variable input controlled by external requests. Direct VLA usage is dangerous on devices with tiny memory stacks.
**Prevention:** Avoid VLAs dependent on unbounded sizes. Strictly prefer checked dynamic allocation (like `malloc` verifying limits) or fixed-size bounded buffers. Explicitly check for `NULL` from `malloc` to handle exhaustion securely without crashing.
